## 1. 사업의 본질 — 두 개의 축

### 우리가 해결하는 문제

경매는 시세보다 저렴하게 부동산을 취득할 수 있는 합리적인 시스템이다. 그러나 두 가지 병목이 참여를 막는다.

**병목 1 — 물리적 제약**: 입찰일에 반드시 법원에 직접 가야 한다. 평일 오전, 수표 발행, 서류 준비, 대기 — 직장인이 반차 이상을 써야 한다.

**병목 2 — 정보 비대칭**: "이 물건을 사도 되는가?"를 판단할 재료가 부족하다. 권리분석은 어렵고, 시세 비교는 번거롭고, 수익 시뮬레이션은 직접 해야 한다.

경매퀵은 이 두 병목을 해결하되, **명확한 우선순위**가 있다.

### 제1축 (메인): 입찰 대리 접수 시스템

**이것이 사업의 기둥이다.** 고객이 법원을 선택하고, 사건번호를 입력하면, 대법원 데이터와 매칭하여 실제 물건을 확인하고, 바로 입찰 대리를 신청하고, 결제하고, 진행 상황을 추적하고, 결과를 통보받는 — 이 전체 플로우가 정확하게 작동하는 것. 이것이 안 되면 나머지는 전부 의미 없다.

핵심 플로우:
```
법원 선택 → 사건번호 입력 → 실제 물건 매칭 확인 → 예약 가능 여부 확인
→ 입찰가·입찰인 정보 입력 → 서류 업로드(전자서명) → 보증금 송금
→ 접수 확정 → 입찰 수행 → 결과 통보 → 정산(낙찰 시 성공보수 / 패찰 시 보증금 반환)
```

**1물건 1고객 원칙**: 같은 회차의 같은 물건에 대해 중복 접수를 받지 않는다. 한 물건에 두 고객의 입찰을 동시에 대리하면 이해충돌이 발생하므로, 이것은 신의성실의 원칙에 기반한 운영 규칙이다. 시스템에서 **사건번호+회차 단위로 물건 상태를 관리**해야 한다: "접수 가능" / "예약됨" / "입찰 완료" / "결과 통보 완료".

### 제2축 (보조): 콘텐츠 허브 — 신뢰와 체류

콘텐츠(물건분석, 경매 가이드, 시황, 뉴스)는 제1축 위에 쌓이는 신뢰와 체류의 레이어다. "여기는 경매에 대해 잘 아는 곳이고, 읽을거리가 있고, 믿을 만하다"는 인상을 주는 역할이다.

바토너에는 이 레이어가 전혀 없다. 바토너의 첫 화면은 "사건번호를 입력하세요"이다. 사건번호가 뭔지 모르는 사람(Segment B)은 이탈한다. 경매퀵은 콘텐츠로 이 사람들을 유입시키고, 콘텐츠를 읽는 과정에서 "여기서 대리입찰도 해주네?"를 발견하게 한다.

**두 축의 관계**: 제1축이 기둥, 제2축이 지붕. 기둥 없이 지붕만 올리면 무너진다. 접수 시스템이 정확하게 작동하는 것이 먼저, 콘텐츠는 그 위에 쌓는 것.

### 데이터 원칙

**원소스: 대법원 공식 경매정보.** 모든 데이터베이스는 대법원이 제공하는 공식 정보를 근간으로 한다.

- **대법원 경매정보** (courtauction.go.kr / 사법정보공유포털 API): 사건 정보, 물건 정보, 입찰 기일, 감정가, 최저가, 등기 정보 등의 원소스.
- **두인옥션 PDF**: 원소스가 아니다. 대법원 정보를 1차 가공한 유료 서비스이며, 현재 API 연동이 구현되지 않은 상태에서 **교차 검증 도구**로 사용하고 있다. API 연동이 완료되면 두인옥션 의존도는 제로에 수렴한다.
- **네이버 부동산 크롤러**: 시세 비교를 위한 보조 데이터. 원소스가 아님.
- **국토부 실거래가**: 실거래 데이터 보조 소스.

대외 표기는 항상 "대법원 경매정보를 기초로 작성"이다. 두인옥션의 이름·구조·표현법은 외부에 노출하지 않는다.

### 수익 모델

| 항목 | 금액 | 조건 |
|------|------|------|
| 얼리버드 수수료 | 5만원 | 입찰일 7일+ 전 신청 |
| 일반 수수료 | 7만원 | 입찰일 2~7일 전 신청 |
| 급건 수수료 | 10만원 | 입찰일 2일 이내 신청 |
| 낙찰 성공보수 | +5만원 | 낙찰된 경우에만 |

**핵심 구조**: 기본 수수료는 결과와 무관하게 선납. 성공보수는 낙찰 시에만. 패찰 시에는 기본 수수료만 받고 보증금은 전액 반환. 이 구조가 가격표(`#pricing`), 신청 페이지(`/apply`), FAQ, 마이페이지에 일관되게 반영되어야 한다.

**"패찰 시 보증금 전액 반환"은 신뢰의 핵심이다.** 이 문구는 다음 위치에 반드시 포함한다:
- `#pricing` 섹션 하단 (가격표 바로 아래)
- `/apply` 신청 페이지 (보증금 안내 영역)
- `/faq` (보증금 관련 질문)
- CTA 다크 블록 또는 신뢰 뱃지 영역
- 마이페이지 접수 상세 화면

### 히어로 = 서비스 인터페이스

히어로는 단순한 헤드라인+검색바가 아니다. **스카이스캐너, 쏘카처럼 고객이 원하는 것을 바로 찾고, 비용을 즉시 확인할 수 있는 서비스 인터페이스**여야 한다.

경매퀵의 비즈니스가 "경매 물건에 대한 입찰 대리"이므로, 히어로는 "어떤 물건을 대리입찰할 것인가"를 바로 탐색할 수 있는 도구여야 한다:

- **법원 선택** (인천지방법원 / 향후 수원, 대전 등)
- **사건번호 직접 입력** (Segment A: 사건번호를 이미 아는 고객)
- **물건 유형·예산 범위 필터** (Segment B: 탐색하는 고객)
- **검색/탐색 실행** → 물건 매칭 확인 → 바로 신청 가능

이 인터페이스 옆에 수수료가 즉시 보여야 한다. "5만원부터"라는 가격이 물건 탐색과 동시에 시각화되어, 고객이 "이 물건을 이 가격에 대리입찰할 수 있다"를 직관적으로 이해하는 구조.

**Phase 1에서의 구현**: 필터 UI는 있되, 검색 결과는 정적 데이터(마크다운 frontmatter 기반 필터링)로 처리. Phase 2에서 대법원 API 연동 시 실시간 검색으로 교체.

### 로그인 후 사용자 경험 (마이페이지)

제1축(접수 시스템)이 제대로 작동하려면, 고객이 자신의 접수 내역과 진행 과정을 확인할 수 있는 로그인 후 경험이 필수다.

**마이페이지 핵심 기능:**
- **현재 접수 현황**: 접수된 물건의 진행 상태 (접수 확인 → 서류 검토 → 보증금 확인 → 입찰 수행 → 결과 통보 → 정산 완료). 각 단계별 타임스탬프와 상태 표시.
- **과거 이용 기록**: 지금까지 의뢰한 모든 건의 이력. 물건 정보, 입찰가, 결과(낙찰/패찰), 정산 내역.
- **내 정보 관리**: 연락처, 알림 설정, 서류(전자본인서명확인서) 관리.
- **보증금 내역**: 입금/반환/법원 납부 이력. 전용계좌 입금 확인 상태.
- **알림**: 접수 확정, 입찰 완료, 결과 통보 등 주요 이벤트 알림 (카카오톡 + 사이트 내 알림).

**Phase 1에서의 구현**: 마이페이지 UI 중 "접수 완료 확인 화면"과 "접수 내역 조회"를 구현한다. 로그인은 Phase 1에서는 간이 인증(이메일+접수번호 조회) 또는 소셜 로그인(카카오) 중 빌드 시점에 판단. 전체 마이페이지(실시간 상태 추적, 보증금 내역, 알림)는 Phase 2에서 백엔드 연동과 함께 완성한다.

### 고객 지원 체계

고객이 서비스를 이용하면서 생기는 의문과 불안을 해소하는 구조가 필요하다.

- **FAQ**: 가장 빈번한 질문을 카테고리별로 정리. 보증금, 서류, 결과, 수수료, 환불 등.
- **1:1 상담**: 카카오톡 채널. 접수는 웹에서 완결하되, 접수 후 확인·문의·결과 통보 등 커뮤니케이션은 카카오톡으로 진행한다.
- **신청 가이드**: `/apply/guide` — 처음 이용하는 고객을 위한 단계별 안내. 스크린샷+설명.
- **상태별 안내**: 접수 후 "다음에 무엇을 해야 하는지"가 명확해야 한다. 각 단계 전환 시 안내 메시지.

**Phase 1에서의 구현**: FAQ 페이지, 신청 가이드 페이지, 웹 접수 폼을 구현. 접수 후 커뮤니케이션(확인, 문의, 결과 통보)은 카카오톡 채널. 마이페이지 내 실시간 상태 추적은 Phase 2.

### 이 이해가 코드에 미치는 영향

- **히어로의 검색/필터 컴포넌트는 별도 파일로 분리**한다. Phase 1에서는 프론트엔드 필터링 + 라우터 이동, Phase 2에서는 API 호출로 교체. `onSearch` 핸들러를 props로 받는 구조.
- **물건 상태 관리 로직은 Phase 1에서도 데이터 구조를 미리 설계한다.** `{caseNumber, round, status, customerId, timestamps}` — Phase 1에서는 구글 시트나 JSON으로 수동 관리하더라도, Phase 2에서 DB로 전환 가능하도록.
- 물건분석 상세 페이지(`/analysis/:slug`)의 하단에는 반드시 "이 물건 입찰 대리 신청" CTA가 있어야 한다. 콘텐츠에서 서비스로의 전환 경로가 끊기면 사업 모델이 작동하지 않는다.
- 홈 페이지의 물건분석 캐러셀은 단순한 콘텐츠 목록이 아니라, "이런 분석을 무료로 해준다"는 가치 시연이다. 카드의 정보 밀도(가격, 태그, 위험도)가 충분해야 한다.
- **대시보드 카드(이번 주 입찰 N건 등)도 Phase 2에서는 API 연동 대상.** Phase 1에서는 수동 업데이트하되, 데이터 소스만 교체하면 되는 구조로 만든다.
- **사이트맵에 마이페이지 관련 경로 추가 필요**: `/my`, `/my/orders`, `/my/history`, `/my/settings` — Phase 2 구현이지만 BUILD_GUIDE에서 경로를 예약해 둔다.

## 2. 사이트의 본질 — 랜딩이 아닌 콘텐츠 허브

### 핵심 인식

경매퀵 웹사이트는 랜딩 페이지가 아니다. **콘텐츠 허브**다.

바토너는 "입찰 신청 도구"다. 사건번호를 넣고 결제하고 나가는 곳이다. 머물 이유가 없다.
경매퀵은 "경매 정보를 읽으러 오는 곳"이다. 물건분석, 경매 가이드, 시황, 뉴스 — 읽을거리가 쌓여 있어서 방문할 때마다 새로운 정보가 있는 곳. 그리고 그 와중에 "여기서 대리입찰도 해주네?"가 자연스럽게 발견되는 구조.

이것이 바토너와의 근본적 차이이며, Segment B를 유입시키는 유일한 방법이다.

### 콘텐츠 유형 (4종)

| 유형 | 발행 주기 | 설명 | URL 패턴 |
|------|----------|------|----------|
| 물건분석 | 수시 (주 3~5건) | 개별 경매 물건의 7섹션 심층 분석 | `/analysis/{slug}` |
| 경매가이드 | 비정기 | 경매 기초~실전 교육 콘텐츠 | `/guide/{slug}` |
| 시황·뉴스 | 주간/월간 | 인천 경매 동향, 낙찰가율 변동, 시장 분석 | `/news/{slug}` |
| 공지·업데이트 | 비정기 | 서비스 변경, 지역 확장, 이벤트 | `/notice/{slug}` |

물건분석이 킬링 콘텐츠다. 나머지 3종은 사이트를 "살아있는 곳"으로 만드는 보조 콘텐츠. 물건분석만 있으면 "분석 모음집"이지만, 시황과 가이드가 함께 있으면 "경매 정보 허브"가 된다.

### 이 이해가 코드에 미치는 영향

- **홈 페이지는 "최신 콘텐츠의 쇼케이스"여야 한다.** 물건분석 캐러셀뿐 아니라, 최신 가이드·시황도 홈에서 미리 보여야 한다. 방문할 때마다 새로운 것이 있어야 한다.
- **네비게이션은 콘텐츠 중심이다.** "물건분석 · 경매가이드 · 시황" 순서. "서비스안내"는 그 다음.
- **사이드바 또는 플로팅 영역에 "최신 글", "인기 분석" 등을 넣어서 탐색을 유도한다.** 단일 글을 읽고 나가는 게 아니라, 다음 글로 이어지는 구조.

## 3. 콘텐츠 관리 구조 — 편집과 표시의 분리

### 핵심 원칙: 워드프레스처럼

물건분석 콘텐츠를 "코드를 수정해서 올리는 것"이 아니라, **마크다운 파일 하나를 추가하면 자동으로 페이지가 생성되는 구조**여야 한다.

```
[내부 편집 형식]                    [외부 렌더링]
content/                           웹사이트
├── analysis/                      ├── /analysis (목록 → 카드 그리드)
│   ├── 2026-04-18-deoksan.md     │   └── /analysis/deoksan-302 (상세 → 7섹션 레이아웃)
│   ├── 2026-04-25-haim.md        │
│   └── 2026-05-02-sewon.md       │
├── guide/                         ├── /guide (목록 → 카드 리스트)
```

### 마크다운 frontmatter가 모든 것을 결정한다

물건분석 마크다운의 상단에 메타데이터(frontmatter)를 넣으면, 빌드 시 목록 카드·상세 페이지·메타 태그·JSON-LD가 자동 생성된다.

```yaml
---
type: analysis              # analysis | guide | news | notice
slug: deoksan-302
title: 덕산하이츠 302호
subtitle: 최저가 9,947만원, 함정은 어디에?
category: danger             # danger | safe | edu
region: incheon
caseNumber: 2021타경521675
appraisal: 203000000
minPrice: 99470000
round: 4
percent: 49
bidDate: 2026-04-18
address: 인천광역시 미추홀구 주안동
propertyType: 다세대주택
area: 49.17
tags: [임차보증금 인수, 강제경매]
publishedAt: 2026-04-15
updatedAt: 2026-04-15
status: published            # draft | published | archived
---

(여기부터 7섹션 본문 — 마크다운으로 작성)
```

### 이 구조의 장점

1. **콘텐츠 추가 = 마크다운 파일 1개 추가.** 코드 수정 불필요.
2. **목록/카드는 frontmatter에서 자동 생성.** 같은 데이터로 홈 캐러셀, 목록 페이지, 상세 페이지가 모두 렌더링된다.
3. **status 필드로 발행 관리.** draft면 빌드에서 제외, published면 노출, archived면 목록에서 숨기되 URL은 유지.
4. **검색·필터가 frontmatter 기반.** 지역별, 카테고리별, 날짜별 필터링이 데이터베이스 없이 가능.
5. **Phase 2 전환 시 마크다운 → DB 마이그레이션이 자연스럽다.** frontmatter가 곧 DB 스키마.

### 기술적 구현 방향

Next.js의 경우 `contentlayer` 또는 `@next/mdx` + `gray-matter` 조합. Astro의 경우 Content Collections. 기술 선택은 BUILD_GUIDE에서 결정하되, 원칙은 동일하다: **마크다운 파일 하나 = 페이지 하나.**

## 4. 고객과 그들의 여정

### Segment A: 경매 경험자

- 사건번호를 이미 알고 있다.
- "법원 안 가고 싸게 입찰 대리하는 곳"을 찾고 있다.
- 사이트에서 원하는 것: **검색 → 가격 확인 → 바로 신청**. 3클릭 이내.
- 설득이 필요 없다. 서비스 존재 자체가 전환 포인트.
- 이 사람을 위한 페이지: 히어로 검색바, `/apply`, 가격표(`#pricing`).

### Segment B: 재테크 관심자

- 사건번호가 뭔지 모른다. 경매 자체가 처음이다.
- 네이버에서 "인천 경매", "빌라 경매 분석" 등을 검색하다가 유입된다.
- 사이트에서의 여정: **블로그/물건분석 글 → 다른 글도 읽음 → "여기서 대리입찰도 해주네?" → 서비스 페이지 → 웹 신청 또는 카카오톡 문의**.
- 신뢰 구축에 시간이 필요하다. 콘텐츠의 양과 질이 핵심.
- 이 사람을 위한 페이지: `/analysis`, `/analysis/:slug`, `/guide`, `/news`.

### Segment B의 체류를 늘리는 구조

Segment B는 한 글을 읽고 나가면 전환이 일어나지 않는다. **2~3개 글을 연속으로 읽게 해야** 신뢰가 쌓이고 서비스로 전환된다.

- 물건분석 상세 하단에 "관련 분석"(같은 동네, 비슷한 유형) 카드 3개.
- 가이드 글 하단에 "이 개념이 적용된 실제 물건" 링크.
- 사이드바 또는 글 사이에 "이번 주 인천 경매 요약" 위젯.
- 모든 콘텐츠 페이지에 "물건분석 모아보기" 배너.

### 이 이해가 코드에 미치는 영향

- 모든 페이지에서 CTA("입찰 대리 신청")가 접근 가능해야 한다 (GNB + 모바일 스티키). Segment A가 어떤 페이지에 있든 바로 신청할 수 있어야 한다.
- 물건분석 상세 페이지는 SEO 최적화가 필수. Segment B의 유입 채널이다. 메타 태그, JSON-LD, 의미 있는 URL slug가 중요하다.
- **물건분석 상세 페이지가 사업의 가장 중요한 페이지다.** 홈보다 중요할 수 있다. Segment B는 홈을 거치지 않고 검색에서 바로 상세 페이지로 들어온다.

## 5. 경쟁 구도와 우리의 포지션

### 바토너 (업계 1위, 우리의 벤치마크)

- (주)케이디씨텍, 2023.07 론칭, 전국 230명 대리인, 보증금 누적 2,400억원.
- **강점**: 시스템화된 웹 플로우, 전국 커버리지, 보증 규모.
- **구조적 약점**: ① 콘텐츠 마케팅 부재 (블로그 0, 물건분석 0, 교육 콘텐츠 0) → Segment B 유입 경로 없음. ② 중개 플랫폼이므로 대리인 품질 통제 어려움. ③ 지역 밀착 부재. ④ "입찰 신청 도구"이지 "머물 곳"이 아니다.

### 이 이해가 코드에 미치는 영향

- **바토너를 사이트에서 직접 언급하지 않는다.** 가격, 회사명, 수수료 숫자 모두 노출 금지. "업계 평균 대비 절반 수준"으로 간접 표현. 이 문서의 바토너 정보는 내부 참조용이며 고객 대면 페이지에 노출되어서는 안 된다.
- **비교표**("직접 입찰 vs 경매퀵")는 바토너가 아닌 "직접 입찰"과 비교한다.
- **콘텐츠의 양 자체가 경쟁 무기**이다. 물건분석 10개, 가이드 5개, 시황 4개가 쌓이면 바토너가 따라올 수 없는 Moat가 된다.

## 6. 판단 기준 (새로운 상황에서의 의사결정 원칙)

코드 작성, 텍스트 작성, UI 설계에서 판단이 필요할 때 아래 6가지 기준에 비추어 결정한다. 우선순위는 번호 순서.

### ① 전환 경로가 유지되는가?

모든 페이지에서 "입찰 대리 신청"까지의 경로가 2클릭 이내여야 한다. GNB CTA + 모바일 스티키 CTA가 항상 접근 가능해야 한다. 새 페이지를 추가할 때 이 경로가 끊기지 않는지 확인한다.

### ② 콘텐츠 체류를 늘리는가?

새로운 기능이나 컴포넌트를 추가할 때, "사용자가 이 사이트에 더 오래 머물게 하는가?"를 따진다. 관련 글 추천, 사이드바 위젯, "다음 글" 네비게이션 등이 이에 해당한다.

### ③ Segment A와 B 중 누구를 위한 것인가?

각 페이지, 각 섹션, 각 컴포넌트가 A를 위한 것인지 B를 위한 것인지 명확해야 한다. 둘 다를 위한 것이면 A가 B의 콘텐츠 때문에 방해받지 않아야 한다.

### ④ "실용·합리·확실"과 맞는가?

장식적 요소를 추가하려 할 때 이 기준에 비춘다. 애니메이션이 정보 전달에 도움이 되는가? 아이콘이 의미를 명확히 하는가? 그렇지 않으면 빼라.

### ⑤ Phase 1의 제약 안에서 가능한가?

Phase 1에서는 데이터가 마크다운 파일 기반이다. DB나 API가 필요한 기능을 만들려 하지 마라. 단, Phase 2 전환이 용이하도록 코드 구조는 열어두라 (아래 섹션 8 참조).

### ⑥ 접근성과 법적 경계를 지키는가?

아래 섹션 7(법적 경계)과 섹션 10(접근성)을 참조한다.

## 7. 법적 경계 — 콘텐츠 작성 시 구체적 가이드

### 우리의 업무 범위

공인중개사법에 따른 매수신청 대리(입찰 대리) **단일 업무만 수행**한다. 아래는 하지 않는다:
- 권리분석 (등기부 해석, 인수/소멸 판단의 최종 결론)
- 투자자문 (매수 추천, 수익 보장)
- 명도 (점유자 퇴거)
- 등기 (소유권 이전)

### 콘텐츠별 경계선

| 표현 | 판단 | 이유 |
|------|------|------|
| "이 물건 사세요" | 금지 | 투자자문 |
| "수익이 보장됩니다" | 금지 | 투자권유 |
| "이 물건은 안전합니다" | 금지 | 권리분석 결론 |
| "대항력이 없으므로 전부 소멸됩니다" | 주의 | 법적 판단의 느낌. "대법원 경매정보에 따르면 전부 소멸로 표기되어 있습니다"로 공공 출처 명시 |
| "이 물건의 구조는 이렇고, 숫자는 이렇습니다" | 허용 | 정보 제공 |
| "감정가 2억 300만원, 4차 최저가 9,947만원입니다" | 허용 | 사실 전달 |
| "시세 대비 최저가가 60% 수준입니다" | 허용 | 데이터 비교 |
| "수익 시뮬레이션 결과 연 수익률 약 8%입니다" | 허용 | 참고 수치 제시 (단, "참고 수치이며 보장하지 않습니다" 부기) |

### 물건분석 콘텐츠의 톤 규칙

물건분석 7개 섹션(개요→입찰경과→권리분석→시세비교→수익시뮬→매각사례→종합의견)에서:

- **데이터 먼저, 해석은 뒤에.** "최저가가 9,947만원입니다. 그러나 임차보증금 인수 시 실질 비용은 2.93억입니다." 숫자가 앞, 의미가 뒤.
- **판단은 제시하되 권유는 하지 않는다.** "이 물건은 일반 투자자가 접근하기 어려운 구조입니다." (O) vs "이 물건은 사지 마세요." (X)
- **교훈으로 마무리한다.** 종합의견 섹션의 "배울 수 있는 것" 박스에서 경매 지식을 전달. 투자 권유가 아닌 교육.
- **시세 데이터가 완비된 상태에서만 콘텐츠를 생산한다.** 크롤러 데이터가 확보되지 않은 물건은 발행하지 않는다. 만약 생산 과정에서 시세 미수신이 발생하면 "시세 데이터 미수신" 플래그를 시각적으로 표시하여 불완전 상태임을 명확히 하고, 해당 상태로 발행하지 않는다.

### 컴플라이언스 4항목 (모든 페이지 풋터 필수)

1. **면책고지**: 본 콘텐츠는 대법원 경매정보 및 공공데이터를 기초로 작성된 참고 자료이며, 투자 권유가 아닙니다. 투자 판단에 대한 책임은 본인에게 있습니다.
2. **업무범위**: 본 서비스는 공인중개사법에 따른 매수신청 대리(입찰 대리) 업무만을 수행하며, 권리분석·투자자문·명도 등의 업무는 포함되지 않습니다.
3. **시세정보**: 주변 시세는 네○○ 부동산, 국토부 실거래가 공개시스템 등 외부 데이터를 참조하였으며, 실시간 시세와 차이가 있을 수 있습니다.
4. **전문가 의뢰 권고**: 본 콘텐츠에 포함된 수익 시뮬레이션은 공개된 시세 데이터를 기초로 산출한 참고 수치이며, 실제 투자 진행 시에는 반드시 법무사·변호사 등 전문가의 권리분석과 자문을 거치시기 바랍니다.

## 8. Phase 설계 — 코드가 미래를 막지 않게

### Phase 1 (현재) → Phase 2 전환 기준

사업계획서의 전환 기준: **수요가 공급을 초과하는 상태의 입증. 인당 월 수익 1,000만원.**

실무적 의미: 월 약 140건 이상. 이 시점에서 수동 운영이 고통스러워지고, 시스템화가 필요해진다.

### Phase 1 제약과 Phase 2 준비를 동시에 반영하는 방법

| 기능 | Phase 1 구현 | Phase 2 업그레이드 | 코드 구조 힌트 |
|------|-------------|------------------|--------------|
| 입찰 신청 | 웹 접수 폼 (프론트엔드 완전 구현). 제출 데이터는 이메일/Webhook으로 수동 확인. | PG 결제 연동 + DB 저장 + 실시간 상태 업데이트 | `/apply` 웹 접수 폼. Phase 1에서는 폼 제출 → 확인 화면. Phase 2에서 결제+DB 연동. |
| 물건 검색/필터 | frontmatter 기반 프론트엔드 필터링 | 대법원 API 실시간 사건조회 | 검색/필터 컴포넌트를 별도 파일로 분리. `onSearch` 핸들러를 props로 받는 구조. Phase 1에서는 정적 필터링, Phase 2에서는 API 호출로 교체. |
| 경매 물건 데이터 | 두인옥션 PDF → 마크다운 수동 변환 | 대법원 경매정보 API 자동 수집 | frontmatter 스키마가 곧 DB 스키마. 데이터 소스만 교체하면 되는 구조. |
| 콘텐츠 관리 | 마크다운 파일 기반 (SSG) | CMS 또는 DB + 어드민 | frontmatter + 마크다운 본문 → DB row + rich text로 마이그레이션. |
| 대시보드 숫자 | frontmatter 집계 또는 JSON 수동 | API 연동 (실시간 경매 건수) | 데이터 소스만 교체하면 되는 구조. |
| 시세 데이터 | 네이버 부동산 크롤러 수동 실행 | 자동 스케줄링 + DB 저장 | 크롤러 결과 JSON → Phase 2에서 DB 저장 + 변동 추적. |
| 고객 후기 | 사전 작성 문안 (6개 풀, 랜덤 3개 표시) | 실 후기 수집 | `content/reviews/` 또는 `data/reviews.json`에서 로드. |
| 결제 | 계좌이체 안내 (웹 접수 완료 화면에서 전용계좌 정보 표시) | PG 연동 (카드/간편결제) | Phase 1에서는 수수료 확인 UI + 계좌이체 안내. Phase 2에서 PG 결제 폼 교체. |

### 데이터 소스 연동 로드맵

경매퀵의 궁극적 목표는 대법원이 제공하는 공공 경매정보 API와 직접 연동하여, 실시간 물건 데이터 기반의 서비스를 제공하는 것이다. 이것은 바토너가 이미 구현한 "사건조회" 기능을 넘어서, 물건분석+시세비교+수익시뮬레이션까지 자동화하는 것이다.

**데이터 소스 경로:**

```
Phase 1 (현재)
  경매 물건: 두인옥션 PDF → 수동 파싱 → 마크다운 frontmatter
  시세 데이터: 네이버 부동산 크롤러 → JSON
  ↓
Phase 1.5 (중간 단계)
  경매 물건: courtauction.go.kr 크롤링 또는 CODEF API → 자동 수집 → 마크다운 자동 생성
  시세 데이터: 크롤러 스케줄링 → DB 저장
  ↓
Phase 2 (목표)
  경매 물건: 사법정보공유포털 API (openapi.scourt.go.kr) → DB → 실시간 검색
  시세 데이터: 네이버 부동산 API 또는 국토부 실거래가 API → DB → 자동 비교
  수익 시뮬: 물건 데이터 + 시세 데이터 → 자동 시뮬레이션 엔진
```

**Phase 1에서의 원칙**: API 연동이 아직 없더라도, 데이터 구조(frontmatter 스키마)를 API 응답 스키마와 호환되도록 설계한다. 그래야 Phase 2 전환 시 컴포넌트 변경 없이 데이터 소스만 교체할 수 있다.

### 지역 확장 시 코드 구조

인천 → 수원·대전·부산·대구 확장 시:
- 물건분석 frontmatter에 `region` 필드 (Phase 1에서 미리 포함, 값은 항상 "incheon")
- 대시보드 숫자에 `region` 기준 필터링 구조
- URL 구조: `/analysis?region=incheon` 또는 `/incheon/analysis` (Phase 2에서 결정)
- Phase 1에서는 지역 필터 UI를 구현하지 않되, 데이터 구조에 region을 포함시켜 놓는다.

## 10. 접근성 최소 기준

경매퀵은 30~50대 직장인·투자자가 주 고객이다. 노안이 시작된 사용자, 모바일에서 한 손으로 조작하는 사용자를 배려한다.

### 필수 기준

- **색상 대비**: 텍스트와 배경의 명도 대비 4.5:1 이상 (WCAG AA). 특히 `--g300`(#A1A1AA)을 흰 배경 위 본문 텍스트로 사용하지 않는다 (대비 2.6:1로 미달). 보조 텍스트는 `--g400`(#71717A, 대비 4.7:1) 이상.
- **터치 타겟**: 모바일에서 탭 가능한 모든 요소의 최소 크기 44×44px.
- **focus 스타일**: 키보드 네비게이션 시 현재 위치가 시각적으로 명확해야 한다. `outline: none`만 쓰지 말고, 커스텀 focus ring(`box-shadow: 0 0 0 3px rgba(37,99,235,0.3)`)을 적용한다.
- **이미지 alt 텍스트**: 모든 의미 있는 이미지에 alt 속성. 장식 이미지는 `alt=""`.
- **폼 라벨**: 모든 `<input>`에 연결된 `<label>` 또는 `aria-label`.

### 이것을 지키는 이유

접근성은 법적 의무가 아니라 고객 배려다. 경매 투자자는 대부분 중장년층이며, 작은 글씨와 희미한 색상은 이탈을 유발한다. 바토너도 이 수준은 맞추고 있다.

## 11. 블로그 ↔ 웹사이트 콘텐츠 전략

### 원본과 요약의 구분

| | 웹사이트 (`/analysis/{slug}`) | 네이버 블로그 |
|---|---|---|
| 역할 | **원본** (풀 7섹션) | **요약 + 유입 유도** |
| 내용 | 전체 분석 (시세, 수익 시뮬, 매각사례 포함) | 핵심 3~4섹션 요약 + "전체 분석은 여기서" 링크 |
| SEO | 구글/네이버 웹 검색 대응. canonical URL 지정. | 네이버 블로그 검색 대응. |
| 목적 | 체류, 신뢰 구축, 서비스 전환 | 유입, 인지도 확보 |

### 이 구분의 이유

- 네이버 블로그에 풀 콘텐츠를 올리면 우리 사이트에 올 이유가 없다.
- 블로그는 "맛보기"로, 사이트는 "전체 식사"로 분리해야 트래픽이 사이트로 흐른다.
- canonical URL은 웹사이트 페이지에 걸어서 SEO 가치가 우리 사이트에 누적되게 한다.

## 12. 디자인 의사결정의 맥락

이 섹션은 디자인 반복 과정(v3→v9.1)에서 확정된 결정과 그 이유를 기록한다. Claude Code가 같은 실수를 반복하지 않게 하는 것이 목적.

### 왜 블루인가

바토너도 블루이지만, 바토너의 블루는 진한 네이비. 우리는 밝은 블루(`#2563EB`)로 차별화. 블루를 버리고 그린이나 코랄로 갈 것도 검토했지만(v5에서 그린 시도 — "약한 느낌"이라는 피드백), 최종적으로 "블루 메인 + 노랑·빨강 키컬러" 조합이 가장 촌스럽지 않고 신뢰감을 주는 것으로 확정.

### 왜 Noto Sans KR인가

초기 DESIGN.md에서는 Pretendard를 권장했으나, 빌드 편의성을 위해 Noto Sans KR로 전환. 이유: Google Fonts에서 바로 로드 가능, 별도 폰트 파일 호스팅 불필요, 빌드 의존성 최소화. Pretendard가 더 세련되지만, Phase 1에서는 속도가 우선.

### 왜 가로 캐러셀인가

물건분석 카드가 세로 리스트로 나열되면 페이지가 길어지고 히어로 아래 콘텐츠가 밀린다. 가로 스와이프로 컴팩트하게 보여주면서 탐색의 재미를 준다.

### 디자인 반복에서 확인된 원칙

- **컬러를 3색 이상 동시에 강하게 사용하면 촌스러워진다.** 블루 메인, 나머지는 키컬러로 최소 사용.
- **오렌지는 블루와 조합 시 촌스러운 결과를 낸다.** 이 프로젝트에서 오렌지를 사용하지 않는다.
- **가격 강조는 별도 배너 박스가 아닌 헤드라인 텍스트로.** "5만원"은 히어로 헤드라인에서 충분히 강조된다.
- **아이콘 배경색은 블루 단색으로 통일.** 블루/옐로/레드를 아이콘마다 다르게 쓰면 산만해진다.
- **서체는 Noto Sans KR.** Google Fonts 로드 편의성 우선.

- **날짜·시간**: 서버 사이드 날짜/시간 생성 시 반드시 `src/lib/datetime.ts` 유틸 사용. `new Date().toISOString()` 직접 호출 금지. 모든 날짜 처리는 Asia/Seoul 기준. (Vercel 서버는 UTC라 한국 새벽 0~9시 KST 시각이 전날로 기록되는 버그 발생)

### 내부 자료 보호

아래 자료는 내부 작업용이며, 원본 구조·표현 방식·데이터 형식이 외부에 노출되어서는 안 된다:

- **바토너 경쟁사 분석**: 가격, 대리인 수, 보증금 누적액 등 모든 수치가 내부 참조 전용.

1. **플랜 먼저, 코드 나중.** 코드 작성 전에 어떤 파일을 만들고, 어떤 컴포넌트를 구현할지 계획을 보여주고 승인을 받는다.

- 데이터 원칙: 원소스 = 대법원 공식 경매정보. 두인옥션 PDF = 교차 검증 도구(원소스 아님). API 연동 완료 시 두인옥션 의존도 제로.
- 1물건 1고객 원칙: 동일 회차 동일 물건 중복 접수 금지. 신의성실 원칙. 물건 상태 관리(접수가능/예약됨/입찰완료) 필수.
- 히어로 = 서비스 인터페이스: 법원 선택 → 사건번호 입력 또는 필터 탐색 → 물건 매칭 → 바로 신청.
- 근거: 기존 설계가 콘텐츠 쪽에 치우쳐 있었음. 접수 플로우가 사업의 메인이라는 우선순위 명확화.
